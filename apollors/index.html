ApolloRS

ApolloRS is my project where I am trying to understand Apollo 11 flight software by bringing it into Rust.

The repository is here: https://github.com/simransummermalik/ApolloRS

Why I chose this project

I chose ApolloRS because Apollo 11 software is one of those projects that feels almost unreal when you actually think about it. It is not just old code. It is code that helped run a spacecraft, using a computer with very limited memory, strange arithmetic, bank switching, interrupts, timing constraints, and a whole style of programming that is very different from how modern software is written.

I liked the idea of taking something historically important and trying to understand it from the inside instead of only reading about it. Rewriting or modeling Apollo software in Rust forces me to slow down and ask what the original machine was actually doing.

Why this is not just "assembly to Rust"

Assembly to Rust is not like translating Python into Java. Assembly is tied very closely to the machine. It uses registers, memory addresses, jumps, banks, and exact instruction behavior.

So for ApolloRS, I am not trying to immediately turn Apollo code into pretty modern Rust. That would be cute, but it would also be kind of fake if I could not prove it behaves like the original.

Instead, I am thinking about the project in layers.

Layer 1: Emulate the Apollo Guidance Computer

This is the safest first layer. The Apollo assembly stays Apollo assembly. Rust acts like the Apollo Guidance Computer and executes the original machine words. 

Apollo assembly
      |
      v
yaYUL assembler
      |
      v
15-bit AGC machine words
      |
      v
Rust AGC emulator
      |
      v
behavior similar to what the original computer would do

This layer matters because I need something faithful before I can make anything readable.

Layer 2: Faithful Rust translation

The second layer is a more direct Rust translation. Each Apollo instruction becomes a Rust operation. This still keeps the program counter, registers, memory, labels, banks, and addresses.

It would not look like normal Rust yet. It would still behave like assembly, just represented through Rust code. But that is useful because I can compare it instruction by instruction against the emulator.

Layer 3: Readable Rust reconstruction

This is the harder and more interesting layer. Eventually, I want parts of the code to become readable Rust: structs, functions, loops, and named guidance logic.

The problem is that the original Apollo assembly does not hand you neat functions like update_landing_state(). It gives you labels, jumps, memory locations, fixed-point math, temporary storage, and comments from a completely different programming world.

So the readable version has to be earned slowly. I do not want to make it pretty before I know it is correct.

Why Apollo is weird

Apollo software has a lot of machine details that normal Rust does not automatically copy.

  • The AGC uses 15-bit words.
  • It uses one's-complement arithmetic.
  • It has positive zero and negative zero.
  • It uses banked memory.
  • Interrupts and timing matter.
  • Some instructions can be modified through indexing.
  • Some code runs through an interpreter inside the AGC itself.

That is why I chose Rust for this. Rust lets me make the weird parts explicit instead of hiding everything inside normal integers. I can create types for AGC words, banks, addresses, traces, and machine state.

What I am doing right now

Right now, ApolloRS is closer to a historical emulator and translation scaffold than a clean rewrite. I am trying to preserve the original source, assemble or load reference rope images, decode AGC machine words, execute them in Rust, and compare behavior carefully.

I also care about provenance. That means keeping track of where the historical source came from, which commit was used, what toolchain assembled it, and what assumptions were made.

My notes describe this direction as keeping the historical Apollo source immutable while building upward from exact word semantics, a lossless parser, audited references, and Rust execution rather than treating a modern rewrite as proof by itself.

The actual goal

The goal is not just to say:

I converted Apollo assembly to Rust.

The better goal is:

I am building several Rust representations of Apollo flight software and checking where their behavior agrees with the original AGC execution.

That means comparing things like the program counter, registers, memory writes, bank state, channel activity, interrupts, guidance variables, and final outputs.

Basically, I want to make Apollo code easier to study without sanding off everything that made it Apollo code.
